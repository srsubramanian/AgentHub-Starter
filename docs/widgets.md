# Widgets

The canvas widget system. Widgets are first-class entities the agent
emits to render UI in the right-hand canvas (or inline next to a chat
message). They're declarative, typed, and have a lifecycle.

## Widget types

Six types defined in `agent/agent/widgets.py`:

| Type | Purpose | Component |
|------|---------|-----------|
| `summary_card` | Key/value pairs with optional trend indicators | [`summary-card-widget.tsx`](../app/(app)/chat/_components/widgets/summary-card-widget.tsx) |
| `results_table` | Tabular data with streaming rows | [`results-table-widget.tsx`](../app/(app)/chat/_components/widgets/results-table-widget.tsx) |
| `timeseries_chart` | Multi-series line chart over time | [`timeseries-chart-widget.tsx`](../app/(app)/chat/_components/widgets/timeseries-chart-widget.tsx) |
| `log_tail` | Auto-scrolling log lines with severity | [`log-tail-widget.tsx`](../app/(app)/chat/_components/widgets/log-tail-widget.tsx) |
| `confirmation` | Yes/no prompt with action buttons | [`confirmation-widget.tsx`](../app/(app)/chat/_components/widgets/confirmation-widget.tsx) |
| `query_plan` | Editable Logs Insights query (placeholder) | not yet implemented |

## Common base

All widgets share these fields:

```python
class BaseWidget(BaseModel):
    id: str                    # ULID, generated by the agent
    type: WidgetType           # discriminator
    placement: "canvas" | "inline"
    status: "draft" | "submitted" | "running" | "complete" | "error" | "cancelled"
    title: str
    created_at: str            # ISO 8601
    created_by_run: str
    error_message: str | None
```

## Lifecycle

Status flow:

```
draft         → user-editable, not yet submitted
submitted     → user approved, agent will execute
running       → execution in progress (e.g. AWS API call)
complete      → finished successfully
error         → finished with an error (error_message populated)
cancelled     → user rejected or aborted
```

Most widgets go straight to `complete`. The HITL pattern (Phase 3+ in
`PLAN.md`) uses `draft → submitted → running → complete`.

## How widgets get to the browser

Each widget tool calls `get_stream_writer()` and writes a `CustomEvent`:

```python
from langgraph.config import get_stream_writer
from agent.events import widget_create_event
from agent.widgets import SummaryCardWidget

writer = get_stream_writer()
widget = SummaryCardWidget(id=str(ULID()), title="...", items=[...])
writer(widget_create_event(widget))   # → CustomEvent(name="widget_create", value=widget.dict())
```

The event flows through:

```
LangGraph stream_writer
  → graph.astream(stream_mode="custom") yields the event
  → FastAPI _sse() encodes it as `data: <json>\n\n`
  → Next.js Route Handler proxies the SSE response
  → @ag-ui/client HttpAgent parses it
  → useAgent's onCustomEvent dispatches to reducer
  → React renders the widget in the canvas
```

## Updates via JSON Patch

To update a widget after creation, emit `widget_update`:

```python
from agent.events import widget_update_event

writer(widget_update_event(widget_id, [
    {"op": "replace", "path": "/status", "value": "complete"},
    {"op": "replace", "path": "/rows", "value": new_rows},
]))
```

The frontend applies the patch via `fast-json-patch` in
`agent-reducer.ts`. This is RFC 6902 — `add`, `replace`, `remove`,
`move`, `copy`, `test`.

Real-world example from `aws_tools.py::list_lambda_functions`:

1. Emit `widget_create` for `results_table` with `status="running"`
2. Call boto3
3. Emit `widget_update` patching `status` and appending `rows`

## Adding a new widget type

### 1. Add the Pydantic model

```python
# agent/agent/widgets.py
class GaugeWidget(BaseWidget):
    type: Literal["gauge"] = "gauge"
    value: float
    min_value: float = 0
    max_value: float = 100
    unit: str = ""
```

Add `"gauge"` to the `WidgetType` Literal and `GaugeWidget` to the
`Widget` union.

### 2. Mirror in TypeScript

```ts
// lib/widgets.ts
export type WidgetType = ... | "gauge";

export interface GaugeWidget extends BaseWidget {
  type: "gauge";
  value: number;
  min_value: number;
  max_value: number;
  unit: string;
}

export type Widget = ... | GaugeWidget;
```

### 3. Build the React component

```tsx
// app/(app)/chat/_components/widgets/gauge-widget.tsx
"use client";
import type { GaugeWidget as GaugeWidgetType } from "@/lib/widgets";

export function GaugeWidget({ widget }: { widget: GaugeWidgetType }) {
  // ... render an SVG gauge or progress bar
}
```

### 4. Register it

```tsx
// app/(app)/chat/_components/widget-registry.tsx
case "gauge":
  return <GaugeWidget widget={widget} />;
```

### 5. Create a tool that emits it

```python
# agent/agent/tools/widget_tools.py
@tool
def create_gauge(title: str, value: float, max_value: float = 100, unit: str = "") -> str:
    """Create a gauge widget showing a single numeric value."""
    writer = get_stream_writer()
    widget = GaugeWidget(id=str(ULID()), title=title, value=value, max_value=max_value, unit=unit)
    writer(widget_create_event(widget))
    return f"Created gauge '{title}'"

WIDGET_TOOLS = [create_summary_card, ..., create_gauge]
```

That's it. Restart the agent; the LLM will see the new tool and use it
when appropriate.

## Why this design?

- **Declarative** — agent describes WHAT to show, frontend decides HOW
- **Typed** — Pydantic + TS catch shape mismatches at boundaries
- **Streamable** — widgets can be created in `running` state and
  updated as data arrives (great for slow APIs)
- **Composable** — multiple widgets per response, ordered by creation
- **Resilient** — error boundary per widget, JSON Patch for partial
  updates, graceful AccessDenied surfaces in the widget itself

## Pitfalls

- **TS / Pydantic drift** — `lib/widgets.ts` is hand-mirrored. Always
  edit both in the same commit.
- **JSON Patch path syntax** — `/items/0/value` not `/items[0].value`.
- **Widget IDs must be unique** — use ULIDs (already the convention).
- **Don't mutate widget state from the frontend.** All updates flow
  from the agent via `widget_update` events.
