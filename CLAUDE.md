# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A starter agentic application demonstrating AG-UI protocol + LangGraph + AWS Bedrock with a chat-plus-canvas UI, declarative widgets, MCP servers, and Anthropic-style Skills. Reference use case: cloud operations assistant.

Detailed documentation lives in [`docs/`](./docs/README.md). Read those before making non-trivial changes — especially `docs/architecture.md`, `docs/widgets.md`, `docs/tools.md`, `docs/mcp-servers.md`, and `docs/skills.md`.

## Repo layout

Single project, not a monorepo. Next.js app at the root, Python agent in `agent/`. No workspaces.

## Critical conventions

- **Widget schema source of truth is `agent/agent/widgets.py`.** TypeScript types in `lib/widgets.ts` mirror it by hand. If you change one, change the other in the same commit.
- **Bedrock model IDs use cross-region inference profiles** (e.g. `us.anthropic.claude-sonnet-4-5-...`), read from `BEDROCK_AGENT_MODEL` env var.
- **AWS auth uses boto3's default credential chain.** `~/.aws/credentials` is mounted read-only into the agent container by docker-compose. Don't hard-require env vars in code.
- **The LLM never formats AG-UI events directly.** Tools emit `CustomEvent` objects via LangGraph's `StreamWriter` (`get_stream_writer()` inside a node). Helpers in `agent/agent/events.py`.
- **Widget IDs are ULIDs**, generated by the agent (`from ulid import ULID`).
- **Widget updates use JSON Patch** (RFC 6902, via `jsonpatch` on Python and `fast-json-patch` on the frontend reducer).
- **Checkpointing is in-memory (`InMemorySaver`).** Postgres is Phase 7. Don't add Postgres deps until then.
- **Tools merge at request time.** `NATIVE_TOOLS` (widget + AWS + skills) plus `_mcp_tools` (loaded async at startup) are combined inside `_all_tools()` in `graph.py` — never bind a stale tool list at module load.
- **MCP config is gitignored.** `agent/mcp_servers.json` may contain secrets; copy from `mcp_servers.json.example`. Use `${VAR}` substitution for tokens, not literals.
- **Skills are SKILL.md files under `agent/skills/<name>/`** with YAML frontmatter (`name`, `description`). Loaded once at startup; advertised in the system prompt; full body fetched via the `invoke_skill` tool when relevant.

## Stack quick reference

- Frontend: Next.js 15 App Router, React 19, TS strict, Tailwind v4, shadcn/ui (New York, Slate), `@ag-ui/client`, `react-markdown` + `rehype-highlight`, recharts
- Backend: Python 3.12, FastAPI, LangGraph + InMemorySaver, `langchain-aws` (ChatBedrockConverse), `ag-ui-protocol`, `langchain-mcp-adapters`, structlog, boto3
- Package managers: npm (frontend), uv (Python)
- Docker compose runs three services: `web` (3000), `agent` (8000), `mcp-http` (8765 — example external MCP server)

## Architecture in one paragraph

User types in chat → `useAgent` hook (`lib/use-agent.ts`) builds an AG-UI `RunAgentInput` and POSTs to `/api/agent/run` → Next.js Route Handler (`app/api/agent/run/route.ts`) is a thin reverse-proxy to FastAPI → `agent/agent/main.py::stream_agent_response` calls `graph.astream(stream_mode=["messages","custom"])` → LangGraph runs `respond → tools → respond` with all tools bound (native + MCP) → `messages` mode emits `TEXT_MESSAGE_*` events for LLM tokens; `custom` mode passes through `widget_create/update/remove` events from tools → SSE flows back to `@ag-ui/client` in the browser → `agentReducer` (`lib/agent-reducer.ts`) updates state → React renders the chat transcript (left, with markdown + syntax highlighting) and the canvas widget stack (right, via the widget registry).

## Code style

- TypeScript: strict mode, no `any`, prefer discriminated unions over flags. Functional components, hooks for state.
- Python: type hints required, mypy strict, Pydantic for all external interfaces, structlog for logging (no `print`).
- shadcn components in `components/ui/` are owned code — edit them when needed.
- Chat components: `app/(app)/chat/_components/` (one file per widget under `widgets/`).
- Custom non-chat components: `components/app/`.

## Things to ask before doing

- Changing the widget schema (cascades through Pydantic + TS + reducer + every component)
- Adding a widget type beyond the existing six
- Adding a dependency not already in `package.json` / `pyproject.toml`
- Touching auth, JWT, or tenant logic (Phase 5 territory — currently stubs)
- Switching from `InMemorySaver` to `PostgresSaver` before Phase 7
- Deleting or renaming SKILL.md files (they're authored content, not generated)

## Things to just do

- Implementing widgets within the existing schema
- Adding new MCP servers via `mcp_servers.json` config (no code changes needed)
- Adding new skills (drop a SKILL.md in `agent/skills/<name>/`)
- Improving styling within the established Tailwind/shadcn tokens
- Refactoring within a phase's scope
- Updating docs in `docs/`

## Workflow

- Work one phase at a time, in order from `PLAN.md`
- Commit at each working milestone — match the existing commit-message style
- Lint/typecheck before every commit
- Update `PROGRESS.md` after each phase

```bash
# Add an MCP server: edit agent/mcp_servers.json, restart agent

```

## Common pitfalls

- **`get_stream_writer()` only works inside a graph node.** Calling it elsewhere raises.
- **Bedrock returns content as a list of typed blocks** (`[{"type":"text","text":"..."}]`), not a string. Use `_extract_text_delta` in `main.py`.
- **`@ag-ui/client` callback names end in `Event`** (e.g. `onTextMessageContentEvent`, not `onTextMessageContent`).
- **`react-resizable-panels` uses `orientation`**, not `direction` — shadcn docs are sometimes outdated here.
- **`AnyMessage` import in `graph.py` must be runtime, not `TYPE_CHECKING`** — LangGraph evaluates `TypedDict` annotations at compile time.
- **MCP loader handles each server independently.** A failing server logs and is skipped — don't add `asyncio.gather` patterns that fail-all.
